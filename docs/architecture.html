Flujo de Autenticacion Authentication Flow

sequenceDiagram participant U as User participant FE as Frontend participant BE as NestJS API participant DB as PostgreSQL U->>FE: Login (email + password) FE->>BE: POST /auth/login BE->>DB: Find user by email DB-->>BE: User record BE->>BE: bcrypt.compare(password, hash) BE->>BE: Verify membership + role BE-->>FE: JWT token + user profile FE->>FE: Store token in localStorage FE->>BE: GET /systems (Authorization: Bearer JWT) BE->>BE: JwtAuthGuard validates token BE->>BE: RolesGuard checks role BE->>DB: Query with organizationId filter DB-->>BE: Tenant-scoped data BE-->>FE: JSON response

Multi-Tenancy

Cada usuario pertenece a una o mas organizaciones via el modelo Membership. El JWT incluye organizationId, y cada consulta a la base de datos filtra por este campo. Esto proporciona aislamiento de datos a nivel de aplicacion. Each user belongs to one or more organizations via the Membership model. The JWT includes organizationId, and every database query filters by this field. This provides application-level data isolation.

Cache con Redis Redis Caching

Redis 7 se usa como capa de cache para reducir carga en PostgreSQL. El backend utiliza Redis para sesiones y cache de respuestas frecuentes (dashboard, health summary, analytics). Redis 7 is used as a caching layer to reduce PostgreSQL load. The backend uses Redis for sessions and caching frequent responses (dashboard, health summary, analytics).
