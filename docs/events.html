Niveles de EventoEvent Levels

Cada evento se clasifica con un nivel de severidad que determina su prioridad visual y las notificaciones que genera. Each event is classified with a severity level that determines its visual priority and the notifications it generates.

⚠
CriticoCritical
critical

Eventos que requieren atencion inmediata. Indican fallas en sistemas productivos, perdida de conectividad, o condiciones que afectan la continuidad del negocio. Generan notificaciones push inmediatas a todos los canales configurados. Events that require immediate attention. Indicate failures in production systems, connectivity loss, or conditions affecting business continuity. Generate immediate push notifications to all configured channels.

Ejemplos: Sistema PRD offline, CPU >95% por 5 minutos, disco lleno, failover de HA activado, certificado SSL expirado. Examples: PRD system offline, CPU >95% for 5 minutes, disk full, HA failover triggered, SSL certificate expired.

⚠
AdvertenciaWarning
warning

Eventos que indican condiciones anormales que podrian escalar a problemas criticos si no se atienden. Generan notificaciones segun la configuracion del escalamiento. Events indicating abnormal conditions that could escalate to critical problems if not addressed. Generate notifications according to escalation configuration.

Ejemplos: CPU >80%, memoria >85%, disco >80%, heartbeat degradado, job fallido en QAS, certificado por expirar en 30 dias. Examples: CPU >80%, memory >85%, disk >80%, degraded heartbeat, failed job in QAS, certificate expiring in 30 days.

ℹ
InformativoInformational
info

Eventos informativos que registran actividades normales del sistema. No generan notificaciones por defecto, pero quedan registrados para auditoria y analisis. Informational events that record normal system activities. Do not generate notifications by default, but are logged for auditing and analysis.

Ejemplos: Runbook ejecutado exitosamente, transporte importado, usuario inicia sesion, configuracion actualizada, health check completado. Examples: Runbook executed successfully, transport imported, user login, configuration updated, health check completed.

✓
ExitoSuccess
success

Eventos que confirman la resolucion exitosa de un problema o la finalizacion de una operacion importante. Utiles para cerrar el ciclo de incidentes. Events that confirm the successful resolution of a problem or the completion of an important operation. Useful for closing the incident cycle.

Ejemplos: Alerta resuelta automaticamente, backup completado, failover exitoso, sistema restaurado, aprobacion concedida. Examples: Alert auto-resolved, backup completed, successful failover, system restored, approval granted.

Modelo de Datos del EventoEvent Data Model

interface Event {
  id:             string;       // UUID v4
  level:          EventLevel;   // critical | warning | info | success
  source:         EventSource;  // SAP | PLATFORM | SECURITY
  component:      string;       // e.g., "HANA", "Runbook Engine", "Auth"
  systemId:       string;       // Related system UUID (nullable)
  sid:            string;       // SID shorthand (e.g., "PRD")
  title:          string;       // Short event title
  message:        string;       // Detailed event description
  metadata:       Record<string, any>; // Additional structured data
  userId:         string;       // User who triggered (nullable for system events)
  organizationId: string;       // Tenant isolation
  timestamp:      ISO8601;      // When the event occurred
  createdAt:      ISO8601;      // When it was recorded
}

Ordenamiento por TimestampTimestamp Ordering

Los eventos se ordenan por defecto en orden descendente por timestamp (los mas recientes primero). El parametro sort permite cambiar el ordenamiento. Events are sorted by default in descending order by timestamp (most recent first). The sort parameter allows changing the ordering. 

// Most recent first (default)
GET /api/events?sort=timestamp:desc

// Oldest first
GET /api/events?sort=timestamp:asc

// By level then timestamp
GET /api/events?sort=level:desc,timestamp:desc
ℹ
Indice de BD: La tabla de eventos tiene un indice compuesto en (organizationId, timestamp DESC) para optimizar las consultas de timeline. Las consultas con filtro por systemId tambien usan un indice dedicado (systemId, timestamp DESC). DB Index: The events table has a composite index on (organizationId, timestamp DESC) to optimize timeline queries. Queries filtering by systemId also use a dedicated index (systemId, timestamp DESC).
