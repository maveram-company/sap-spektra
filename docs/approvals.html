Ciclo de Vida del EstadoStatus Lifecycle

Cada aprobacion sigue un ciclo de vida estricto con transiciones definidas. Una aprobacion no puede volver a un estado anterior. Each approval follows a strict lifecycle with defined transitions. An approval cannot return to a previous state.

Niveles de SeveridadSeverity Levels

Las aprobaciones se clasifican por severidad, lo cual determina el timeout y los requisitos de aprobacion. Approvals are classified by severity, which determines the timeout and approval requirements.

Requisitos de RolRole Requirements

El flujo de aprobaciones implementa separacion de funciones (SoD) para asegurar que quien solicita no es quien aprueba. The approval workflow implements separation of duties (SoD) to ensure that the requester is not the approver.

Quien Crea (Solicita)Who Creates (Requests)

  • operator: Crea solicitudes de aprobacion para operaciones en sistemas asignadosoperator: Creates approval requests for operations on assigned systems
  • Sistema (automatico): Genera aprobaciones cuando un runbook con HUMAN gate se ejecuta en PRDSystem (automatic): Generates approvals when a runbook with HUMAN gate runs on PRD

Quien Aprueba/RechazaWho Approves/Rejects

  • escalation: Puede aprobar/rechazar aprobaciones de severidad normal y highescalation: Can approve/reject normal and high severity approvals
  • admin: Puede aprobar/rechazar cualquier aprobacion, incluyendo criticaladmin: Can approve/reject any approval, including critical
⚠
Separacion de Funciones: Un usuario NO puede aprobar una solicitud que el mismo creo. Esto se valida en el backend y la API retorna 403 Forbidden si se intenta. Separation of Duties: A user CANNOT approve a request they created themselves. This is validated on the backend and the API returns 403 Forbidden if attempted.

Aprobaciones Basadas en TokenToken-Based Approvals

Para aprobaciones urgentes, Spektra soporta aprobacion via token unico enviado por email o Slack. El aprobador puede aprobar/rechazar sin necesidad de autenticarse en la plataforma. For urgent approvals, Spektra supports approval via unique token sent by email or Slack. The approver can approve/reject without needing to authenticate on the platform.

Generacion de Token: Cuando se crea una aprobacion, el sistema genera un token JWT firmado con expiracion igual al timeout de la aprobacion. Token Generation: When an approval is created, the system generates a signed JWT token with expiration equal to the approval timeout.
Envio de Notificacion: El token se incluye en URLs de accion directa enviadas por email/Slack: /approvals/action?token=xxx&action=approve Notification Send: The token is included in direct action URLs sent via email/Slack: /approvals/action?token=xxx&action=approve
Validacion: El backend verifica la firma del token, la expiracion, y que el usuario asociado tiene el rol requerido. Tokens de un solo uso. Validation: The backend verifies the token signature, expiration, and that the associated user has the required role. Single-use tokens.

Aprobaciones Disparadas por RunbooksRunbook-Triggered Approvals

Cuando un runbook con un HUMAN gate se ejecuta en un sistema productivo, el motor de ejecucion automaticamente crea una solicitud de aprobacion y pausa la ejecucion hasta que se resuelva. When a runbook with a HUMAN gate runs on a production system, the execution engine automatically creates an approval request and pauses execution until it is resolved.

Modelo de DatosData Model

interface Approval {
  id:              string;         // UUID v4
  title:           string;         // Short description
  description:     string;         // Detailed description
  type:            ApprovalType;   // RUNBOOK_GATE | OPERATION | MANUAL
  severity:        Severity;       // critical | high | normal
  status:          ApprovalStatus; // PENDING | APPROVED | REJECTED | EXECUTED | EXPIRED
  systemId:        string;         // Target system UUID
  sid:             string;         // SID shorthand
  requestedBy:     string;         // User who created the request
  approvedBy:      string | null;  // User who approved/rejected
  comment:         string | null;  // Approver's comment
  runbookId:       string | null;  // Associated runbook (if RUNBOOK_GATE)
  executionId:     string | null;  // Associated execution (if RUNBOOK_GATE)
  operationId:     string | null;  // Associated operation (if OPERATION)
  token:           string;         // JWT for token-based approval
  expiresAt:       ISO8601;        // When the approval expires
  approvedAt:      ISO8601 | null; // When it was approved/rejected
  executedAt:      ISO8601 | null; // When the operation was executed
  organizationId:  string;         // Tenant isolation
  createdAt:       ISO8601;
  updatedAt:       ISO8601;
}
