Modelo Multi-Modo Multi-Mode Model

La arquitectura central de SAP Spektra: 4 modos, 12 dominios, proveedores intercambiables SAP Spektra's core architecture: 4 modes, 12 domains, interchangeable providers

Principios Principles

  • Contrato unico por dominio:Single contract per domain: Cada dominio define una interfaz TypeScript. Todos los proveedores (real, mock, fallback, restricted) la implementan.Each domain defines a TypeScript interface. All providers (real, mock, fallback, restricted) implement it.
  • ProviderResult<T>: Toda respuesta de proveedor incluye data, source, confidence, degraded, timestamp, y reason opcional.Every provider response includes data, source, confidence, degraded, timestamp, and optional reason.
  • Resolucion automatica:Automatic resolution: El motor de capacidades resuelve el tier de cada dominio segun el modo activo y si el backend esta accesible.The capability engine resolves each domain's tier based on the active mode and backend reachability.
  • Transparencia en UI:UI transparency: Componentes como ModeBadge, SourceIndicator y CapabilityBadge muestran al usuario de donde vienen los datos.Components like ModeBadge, SourceIndicator, and CapabilityBadge show the user where data comes from.
  • Paridad testeable:Testable parity: Parity tests verifican que mock y real proveen la misma forma de datos.Parity tests verify that mock and real provide the same data shape.

ProviderResult<T>

La salida canonica de todo proveedor. Envuelve los datos con metadata sobre su origen. The canonical output of every provider. Wraps data with metadata about its origin. 

interface ProviderResult<T> {
  data: T;                        // The actual domain data
  source: ProviderTier;           // 'real' | 'mock' | 'fallback' | 'restricted'
  confidence: 'high' | 'medium' | 'low';
  timestamp: string;              // ISO 8601
  degraded: boolean;              // true if operating in degraded state
  reason?: string;                // Optional explanation
}
💡
La funcion helper providerResult() resuelve confidence automaticamente: real → high, fallback → medium, mock/restricted → low. The helper function providerResult() resolves confidence automatically: real → high, fallback → medium, mock/restricted → low.

Arquitectura de Proveedores por Dominio Provider Architecture per Domain

Cada dominio sigue esta estructura de archivos: Each domain follows this file structure:

  {domain}/

Testing de Paridad Parity Testing

Cada dominio tiene un archivo *.parity.test.ts que verifica que mock y real proveedores devuelven datos con la misma forma (mismos campos, mismos tipos). Esto garantiza que cambiar de modo no rompa la UI. Each domain has a *.parity.test.ts file that verifies mock and real providers return data with the same shape (same fields, same types). This guarantees that switching modes doesn't break the UI.

Adaptador dataService dataService Adapter

dataService.ts actua como capa de compatibilidad retroactiva. Las paginas importan dataService.getSystems() y obtienen datos ya extraidos (.data) del ProviderResult. Para acceso completo al metadata, usan getSystemsResult(). dataService.ts acts as a backward-compatibility layer. Pages import dataService.getSystems() and get data already extracted (.data) from ProviderResult. For full metadata access, they use getSystemsResult(). 

// Unwrapped (backward-compat):
const systems = await dataService.getSystems();  // SystemViewModel[]

// Full ProviderResult:
const result = await dataService.getSystemsResult();
// result.data: SystemViewModel[]
// result.source: 'real' | 'mock' | 'fallback' | 'restricted'
// result.confidence: 'high' | 'medium' | 'low'
// result.degraded: boolean
